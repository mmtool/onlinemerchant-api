Online Merchant Integration Guide

MoMoney Wallet System API Specification for third‑party Online Merchants

📄 v1.0.4 📅 Feb 2023 🔐 OAuth 2.0 🌐 REST API

Introduction

This document defines the process for a 3rd party Online Merchant to connect to the MoMoney Wallet System. It covers the complete integration lifecycle — from authentication through payment initiation, enquiry, confirmation, refund, and callback handling.

Sandbox
https://test-openapi.momoney.com.mm
Production
https://openapi.momoney.com.mm

Customer Journey

The customer journey follows a sequential flow: the customer initiates a payment → the merchant requests a QR or push payment → the subscriber authorizes the payment in the wallet app → the Wallet System processes the transaction → a callback is sent to the merchant with the payment result.

General flow: Authentication → Payment initiation (QR/Push) → Subscriber payment → Wallet callback → (Optional) Refund

1.1 Get Access Token

This endpoint uses the OAuth 2.0 client_credentials grant to obtain an access token. You will need this token to call all subsequent APIs. The token has a limited lifetime (3600 seconds by default).

POST /oauth2/token

Request Header

FieldValue
Content-Typeapplication/json
AuthorizationBasic {Base64(consumer_key:consumer_secret)}

Request Body

ParameterTypeMandatoryDescription
grant_typeStringRequiredFixed value: client_credentials

Response Parameters

FieldTypeDescription
access_tokenStringToken to access the Wallet System
scopeStringam_application_scope default
token_typeStringBearer
expires_inStringExpiry time in seconds (e.g., 3600)

consumer_secret: TFH1UKf5ADZ6CQUFaJCuICmM1KYa

Host: https://test-openapi.momoney.com.mm
Authorization: Basic {Base64(consumer_key:consumer_secret)}
Content-Type: application/json

{"grant_type": "client_credentials"}

1.2 Get User Token

After obtaining an access token, use this endpoint to log in as a merchant and get a user‑scoped token. This token identifies your merchant session and is required for all payment‑related operations.

POST /partner/merchant/login

Request Header

FieldValue
Content-Typeapplication/json
TokenBearer {Access_token}

Request Body

ParameterTypeMandatoryDescription
phoneStringRequiredMerchant phone number
passwordStringRequiredMerchant password

Response Parameters

FieldTypeDescription
errStringError code; 200 = success
messageStringDescription (e.g., "Success")
token.tokenStringUser token (UUID)
token.expiredAtStringExpiry time (ISO 8601)
Sample Request
POST /partner/merchant/login
Host: test-openapi.momoney.com.mm
Content-Type: application/json
Token: Bearer {access_token}

{"phone":"0950000004","password":"841539"}

    "token": "5e2f768a-5228-49ca-b228-3fc1eaa27373",

2. Request QR Payment

Generate a QR code that the subscriber scans with their wallet app to complete the payment. The response includes both a raw QR string and a Base64‑encoded PNG image, as well as a deep link for in‑app payments.

POST /partner/merchant/getQRPayment

Request Headers

FieldValue
Content-Typeapplication/json
Accept-Languageen (English) or my (Myanmar)
AuthorizationBearer {User_token}
TokenBearer {Access_token}

Request Body

ParameterTypeMandatoryDescription
amountNumberRequiredTransaction amount
externalTransactionIdStringRequiredUnique merchant transaction ID
externalAdditionalDataStringOptionalAdditional bill details
currencyStringRequiredMMK or USD

Response (key fields)

FieldTypeDescription
data.qrdataStringRaw QR code string
data.qrBase64Base64Base64‑encoded PNG image
data.deeplinkStringDeep link for subscriber app
data.referenceNumberStringWallet‑generated reference
data.fees.debitFeeNumberFee charged to customer

4. Request Push Payment

Send a push notification directly to a subscriber’s wallet app, prompting them to approve a payment. This requires the subscriber’s phone number.

POST /partner/merchant/sendPushPayment

Request Body

ParameterTypeMandatoryDescription
amountNumberRequiredTransaction amount
externalTransactionIdStringRequiredUnique merchant transaction ID
customerPhoneStringRequiredSubscriber’s registered phone
currencyStringRequiredMMK or USD

Response

FieldTypeDescription
data.referenceNumberStringWallet‑generated reference
data.externalTransactionIdStringMerchant transaction ID

5. Return Payment Result (Callback)

After the subscriber completes the payment, the Wallet System sends an encrypted result to your registered callback URL. The body contains two fields: paymentResult (AES‑encrypted JSON) and checksum (SHA‑256 of the raw JSON). You must decrypt the payload using the provided key.

Decryption key (sandbox): QEOYZtmGazgtgA27iNy8X4TKrXOiFLsU
Algorithm: AES‑256, mode ECB, padding PKCS7.

Decrypted Result Object

FieldTypeDescription
amountNumberTransaction amount
totalAmountNumberTotal including fees
currencyStringMMK
statusString"done" or "failed"
referenceNumberStringWallet reference number
externalTransactionIdStringMerchant transaction ID
transRefIdStringWallet system transaction ID
customer.nameStringCustomer name
customer.phoneStringCustomer phone
merchant.nameStringMerchant name
fees.debitFeeNumberCustomer fee

7. Enquiry Transaction API

This endpoint must be called before confirming a transaction. It verifies the transaction amount and other details to prevent mismatches. You need a device ID and shop ID.

POST /partner/merchant/enquiryTransaction
Always call this API prior to confirmTransaction.

Additional Headers

FieldValue
DeviceIdUnique device identifier

Request Body

ParameterTypeMandatoryDescription
serviceCodeStringRequiredFixed: QRCODEPAYMENT
amountNumberRequiredTransaction amount
currencyStringRequiredMMK
qrcodeStringRequiredScanned QR code
deviceIdStringRequiredDevice ID
shopIdStringRequiredMerchant shop ID
externalTransactionIdStringRequiredMerchant transaction ID
beneficiaryPhoneStringOptionalConsumer phone
beneficiaryNameStringOptionalConsumer name

8. Confirm Transaction API

Finalize a transaction after a successful enquiry. The request parameters are identical to the Enquiry API. The response includes the final transaction details with fees and discounts.

POST /partner/merchant/confirmTransaction
Sample Response
{
  "err": 200,
  "message": "Success",
  "data": {
    "referenceNumber": "5f2bce3932ac3d36099546fa",
    "amount": 1250,
    "totalAmount": 1250,
    "fees": { "debitFee": 250, "creditFee": 350 },
    "externalTransactionId": "HD100011"
  }
}

9. Request Hosted Form

Request a hosted web payment form URL that can be opened in a mobile browser for the subscriber. Requires an HMAC‑SHA1 signature to ensure integrity.

POST /partner/merchant/requestHostedForm

Signature Generation

Sort parameters alphabetically, concatenate as AMOUNT=1000&CURRENCY=MMK&EXTERNALTRANSID=..., then compute HMAC‑SHA1 with your secret key.

Example
// concatenatedString = AMOUNT=1000&CURRENCY=MMK&EXTERNALTRANSID=sdbfh74b74bdasdbh43vb238
// SIGNATURE = 77D34189B76650CC790025593A68453EBFD19657

Response

FieldDescription
data.webVariableURL to hosted payment form
data.transaction.transRefIdTransaction reference ID
